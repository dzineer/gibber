quick start

Getting Started.

From zero to your first .gibber file in under two minutes.

Step 1

Installation

One command. No dependencies beyond bash and curl.

terminal bash 
curl -sL https://raw.githubusercontent.com/dzineer/gibber/main/gibber-install.sh | bash

What it installs

The spec, dictionary, rules, templates, and tool wrappers. Everything an AI agent needs to read and write Gibber from day one.

Where

~/.claude/shared/gibber/ — a shared location accessible to every Claude Code project on your machine.

Idempotent

Safe to re-run at any time. It checks versions, backs up existing files, and only overwrites when the upstream version is newer.

Step 2

Your first .gibber file

A task file is the simplest Gibber structure. Here is one with annotations.

tasks/current_task.gibber gibber 
---
id: CURRENT
gibber_dict: meta/v2
---
(§task §id:T001 §status:§wip
  §title:"set up project scaffolding"
  §owner:§ai
  §created:2026-04-13
  §goal:"cargo init, add deps, create module stubs"
  §todo:[
    "create Cargo.toml with initial deps"
    "add src/lib.rs module structure"
    "write first test"])

Frontmatter

Every .gibber file starts with a YAML frontmatter block containing id and gibber_dict. The id is a unique identifier; the dict tells the agent which symbol vocabulary to load.

The form

After the frontmatter, the body is a single S-expression form. Parentheses delimit structure. §task is the form type. Fields like §id: and §status: are key-value pairs. Lists use square brackets.

Step 3

Templates

Six standard templates ship with the installer. Copy, fill in the placeholders, and go.

task.gibber

Single task with id, status, title, owner, goal, todo list, and result fields. The workhorse of project tracking.

index.gibber

Task index with active, next, and later queues. Keeps the backlog organized with phase grouping.

completed.gibber

Archive of finished tasks with results and timestamps. Grows over time; agents use it for retrospectives.

learning.gibber

Failure log with context, what was tried, outcome, and lesson learned. Prevents the agent from repeating mistakes.

session.gibber

Session log capturing prompts, responses, decisions, and artifacts. Survives context compaction.

memory.gibber

Persistent memory dump with facts, preferences, and project state. The agent reads this on every session start.

Step 4

Project setup

Add Gibber to any Claude Code project in three steps.

1

Add the Gibber Protocol block to your CLAUDE.md

Copy the Gibber Protocol section from the spec into your project's CLAUDE.md. This teaches every agent session how to read and write Gibber, including the fast-path version check.

2

Create a project extension dictionary

Create gibber-dict-yourproject.md in the project root. Add domain-specific symbols here. Start empty and grow it as needed.

3

Initialize the file structure

Copy templates from ~/.claude/shared/gibber/templates/ into your project's tasks/ and memory/ directories. Fill in the placeholders. The agent will take it from there.

Concept

The two-file model

Every Gibber document exists as two files: the source of truth and its human-readable derivative.

.gibber — source of truth

Written and read by the AI agent directly
Contains the Gibber AST form with optional YAML frontmatter
All edits happen here. This file is always authoritative.
Compact: typically 2–5x fewer tokens than English equivalent

.human — derivative

Generated on demand when a human asks to see the file
Rendered as English Markdown, fully readable without knowing Gibber
Never edited directly — always regenerated from the .gibber source
Ephemeral: may be stale if the .gibber file has changed since last render
