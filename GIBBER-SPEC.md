# Gibber Specification

Version: `gibber/4`

Gibber is a compact symbolic file format for AI agent project files. Files written in Gibber are designed to be read **by AI agents**, not by humans. The grammar is a small S-expression-style notation that uses far fewer tokens than the equivalent English, lets agents hold longer context histories before compaction, and is fully self-describing through a versioned dictionary.

## Why this exists

- **Speed and density for the AI**: agents read gibber files directly. The format is dense, low-token, and uses a closed vocabulary that the agent already knows. Long working sessions hold many more files in context.
- **Build for the AI, not the human reviewer**: as agents grow more capable and trusted, the percentage of time a human looks at intermediate working files drops toward zero. The right design assumption is that the human will not be reading these. We optimize for the agent.
- **Humans get English on demand**: when a human wants to see the contents of a gibber file in plain English, the AI generates a readable view in the conversation, or writes a derivative `.human` companion file. The English view is generated from the gibber file, never the other way around.
- **Portable**: the protocol is small enough to teach an AI agent in a single message. Works in any Claude Code project, in any agentic workflow, on any machine.

## File model

Gibber uses two file extensions with two clear roles:

### `.gibber` — the source of truth, what the AI reads

A `.gibber` file holds a single Gibber AST form, optionally with a tiny YAML frontmatter for the file id and dictionary version. It has **no English section, no `## gibber` heading, no `## human` heading**. It is just the Gibber form. This is the format the agent reads, writes, edits, and reasons about. It is the authoritative source.

```
---
id: T042
gibber_dict: meta/v1, eidetic_memory
---

(§task §id:T042
  §status:§wip
  §depends:[T041 T038]
  §owner:§ai
  §goal:(§build §glp §encoder §with:[§ast §round-trip §bench])
  §files:[crates/eidetic-glp/src/encoder.rs
          crates/eidetic-glp/src/dict.rs]
  §done:[§encode §decode §round-trip-test]
  §todo:[§load §dict §bench §integrate §verifier]
  §budget:(§time<2d §lat<1ms §p99<3ms §acc=100%)
  §tests:[§unit §round-trip §latency])
```

That's the entire file.

### `.human` — derivative, generated on demand, what the human reads

A `.human` file is a plain English Markdown rendering of a `.gibber` file. It is **generated by the agent**, never authored by hand. It exists only when a human has asked to see one (and has chosen to save it to disk rather than just read it in the conversation). It is read-only and ephemeral. The corresponding `.gibber` file is always the source of truth; the `.human` file is just a view.

Rules for `.human` files:

1. The `.gibber` file is the authoritative source. The `.human` file is derived from it via the `render-english` rules in `GIBBER-RULES.md`.
2. Agents never edit a `.human` file directly. To change content, edit the `.gibber` file and re-render.
3. If a `.gibber` file is newer than its `.human` companion, the `.human` is stale and must be regenerated (or deleted) before showing it to a human.
4. `.human` files are generally gitignored. Only `.gibber` files belong in version control.
5. The agent generates `.human` views on demand during a conversation, by running the matching `render-english` rule on the parsed `.gibber` form. No human-rendering step is built into any tool wrapper or any file-write operation; it happens only when a human explicitly asks.

### Compatibility with the older dual-language format

Earlier versions of Gibber used a single Markdown file with `## gibber` and `## human` headings. That model is **deprecated** as of `gibber/4`. The two-file `.gibber` / `.human` model replaces it. Conversion is mechanical: keep the gibber section, drop the human section, rename the file from `.md` to `.gibber`, and remove the `format: dual` frontmatter line.

## Gibber-speak grammar

Gibber-speak is a small S-expression-style notation. The grammar is intentionally tiny so it can be parsed by ~100 lines of code in any language.

### Tokens

- **Symbols** start with `§` followed by lowercase ASCII letters, digits, or hyphens. Examples: `§task`, `§wip`, `§glp-encoder`, `§p99`.
- **Identifiers** are bare alphanumeric strings (no leading `§`). Used for IDs, file paths, names. Examples: `T042`, `crates/eidetic-glp/src/encoder.rs`.
- **Numbers** are bare digits, optionally with a decimal point or unit suffix. Examples: `2d`, `15ms`, `100`, `0.95`.
- **Operators**: `< > = <= >=`. Used in budgets and constraints.
- **Punctuation**: `( ) [ ]` for nesting and lists.
- **Field-value separator**: `:` immediately after a symbol binds a value to it. Example: `§status:§wip`.

### Structures

- **Atom**: any single token. `§wip`, `T042`, `15ms`.
- **List**: `[a b c]` — an ordered, unnamed sequence.
- **Form**: `(§head a b c)` — a named structure where `§head` is the type and the rest are positional or named children.
- **Field**: `§key:value` — binds a value to a field. The value can be an atom, a list, or a nested form.

### A complete task example

```
(§task §id:T042
  §status:§wip
  §depends:[T041 T038]
  §owner:§ai
  §goal:(§build §glp §encoder §with:[§ast §round-trip §bench])
  §files:[crates/eidetic-glp/src/encoder.rs
          crates/eidetic-glp/src/dict.rs]
  §done:[§encode §decode §round-trip-test]
  §todo:[§load §dict §bench §integrate §verifier]
  §budget:(§time<2d §lat<1ms §p99<3ms §acc=100%)
  §tests:[§unit §round-trip §latency])
```

This is one form named `§task` with several fields. The whole structure is a tree. A parser can decode it into a typed struct in any language.

### Whitespace and comments

- Whitespace is insignificant except as a token separator.
- Newlines and indentation are encouraged for readability but not required.
- Comments start with `;` and run to end of line.

## The single-source-of-truth rule

Every `.gibber` file is the **only authoritative** representation of its content. There is no second half, no parallel section, no dual-language pair to keep in sync.

1. **The `.gibber` file is the source.** Always. Edits go here.
2. **`.human` files are derived.** Generated on demand by the agent from the matching `.gibber` file via the `render-english` rules in `GIBBER-RULES.md`.
3. **No drift is possible** because there is only one editable artifact per piece of information.
4. **Validation is single-source.** A validator parses the `.gibber` file's form and applies the matching `validate` rule from `GIBBER-RULES.md`. There is no two-half comparison because there is no second half.

## Dictionaries

A gibber dictionary is a list of symbols and their meanings, written in English. Dictionaries are versioned. A file's frontmatter names which dictionary it uses, e.g. `gibber_dict: meta/v1`.

There are three kinds of dictionaries in the standard Gibber distribution:

1. **Meta-work dictionary** (`meta/v1`) — for task files, learning logs, session logs, memory dumps. Project-agnostic. Lives in `GIBBER-DICTIONARY.md`.
2. **Project-specific extensions** — each project may add domain symbols in its own `gibber-dict-<project>.md` file. The project's `CLAUDE.md` lists which extensions are loaded alongside `meta/v1`.
3. **Specialized dictionaries** — runtime AI-to-AI compression, AST encoding for source code, etc. These are reserved for tools that build on Gibber, not part of the meta-work standard.

## Built-in functions (the standard library)

Gibber includes a small standard library of functions the AI executes mentally on read. They are "functions" only in the sense of pattern-rewrite rules — there is no external code, no library to import, no runtime to install. The AI reads the rule from this spec and applies it whenever it sees the pattern in any gibber file.

These functions exist to keep gibber files dense without losing readability. A repeated structure can collapse into a single function call, and the AI expands it on read. When the AI writes the human half of a dual-language file, it always expands functions into their resolved forms so the user sees the final content with no `§rep` or `§ref` mysteries.

### `§rep` — repetition

```
(§rep N X)  ->  [X X X ... X]   (X repeated N times)
```

Example:
```
(§rep 3 §test)   ->   [§test §test §test]
```

### `§range` — integer range expansion

```
(§range A B)   ->   [A A+1 A+2 ... B]
```

Works for numbered IDs and integers. Example:
```
(§range T001 T003)   ->   [T001 T002 T003]
```

### `§ref` — cross-reference within the same file or task index

```
(§ref ID)         ->  the structure named ID, in full
(§ref ID FIELD)   ->  the value of FIELD within the structure named ID
```

Example, assuming task T042 has `§status:§wip`:
```
(§ref T042)            ->  the entire T042 structure
(§ref T042 §status)    ->  §wip
```

This means a task index does not have to restate every task's full content. It can list IDs and use `§ref` to pull fields when needed.

### `§sub` — substitution macro

```
(§sub NAME VALUE)
```

Defines NAME as VALUE within the scope of the current file. After definition, every appearance of NAME in the same file expands to VALUE. Useful for naming a recurring tuple or list once and reusing the name.

Example:
```
(§sub §my-stack [§rust §kuzu §glp])
...
§my-stack   ->   [§rust §kuzu §glp]
```

### `§expand` — symbol to English

```
(§expand §wip)   ->   "work in progress"
```

Looks the symbol up in the loaded dictionary and emits its English meaning. Used when generating the human half of a dual-language file.

### `§compress` — English to symbol

```
(§compress "work in progress")   ->   §wip
```

The inverse of `§expand`. Used when authoring the gibber half from prose.

### `§concat` — concatenation

```
(§concat A B C)
```

Joins atoms or lists. With strings, produces a single string. With lists, produces a single flattened list.

### `§lookup` — return the dictionary definition string

```
(§lookup §wip)   ->   "work in progress"
```

Like `§expand` but always returns the string definition exactly as written in the dictionary, without context-specific transformation.

### Usage hints (when to use these on write, not just read)

- If you write the same symbol three or more times in a row, use `§rep`.
- If you list a contiguous range of IDs, use `§range`.
- If a task field would restate another task's content verbatim, use `§ref`.
- If a tuple or list appears two or more times in the same file, define it once with `§sub` and reuse the name.
- When writing the human half, expand all function calls into their resolved values so the user sees plain prose.
- Never use a function the AI cannot execute from this spec alone. There is no escape hatch to external code.

## Validation rule

A gibber file is valid if and only if:

1. The frontmatter contains `format: dual` and a `gibber_dict` entry.
2. The body contains exactly one `## gibber` and exactly one `## human` section.
3. The gibber section parses cleanly under the named dictionary (every symbol is in the dictionary).
4. The human section is non-empty plain English Markdown.
5. A semantic comparison of the two sections (ID, status, key fields) finds no contradictions.

If any of these fails, the file is rejected and the agent must fix it before writing.

## How an agent uses gibber

1. **On reading a gibber file**: skip the human section, parse the gibber section into a typed structure, work with the structure.
2. **On writing or updating a gibber file**: produce the gibber form first, then mirror it into English, then write both into the same file in one edit.
3. **On encountering a new symbol the agent wants to use**: check the loaded dictionary. If the symbol is not present, add it to the appropriate extension dictionary (with a clear English definition) before using it. Never use undefined symbols.
4. **On detecting drift between the two sections**: stop, fix the disagreement, then proceed.

## Why this matters for agentic workflows

In a long-running agent project, internal documents (task lists, learning notes, session logs, memory dumps) accumulate fast. They eat context budget. They slow generation. They make session compaction more lossy.

Gibber addresses all three: dense gibber halves stay in the agent's context, English halves stay readable for the human reviewer, and both views live in the same file with no translation step. This is dogfooding compression at the meta-work layer of the project itself.

## Portability

This entire specification, plus the standard meta-work dictionary, plus the templates, fits in a few small files. They are designed to be:

- **Copied into any Claude Code project** with one install command.
- **Pasted into any agent's CLAUDE.md** so the agent learns gibber immediately on its next read of project rules.
- **Shared with another developer** as a single archive or even a single Markdown file containing everything.
- **Extended per project** by adding a small domain-specific dictionary alongside the standard one.

The goal is that any agent in any project can speak gibber from its first message, with no per-project re-teaching.
