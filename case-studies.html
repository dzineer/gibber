Case studies: Gibber under real load.

These are honest, measured evaluations of Gibber's effectiveness in real AI agent workflows. Each study records what worked, what didn't, and the token/time impact compared to plain English.

Case Study 001

Installing OpenVox TTS with Gibber task tracking

A five-task Docker-based install of a self-hosted text-to-speech system, managed end-to-end with Gibber task files. Single session, single agent, real project.

2026-04-11 Claude Opus 4.6 5 tasks Single session Docker install

The project

OpenVox is a self-hosted podcast studio and OpenAI-compatible TTS server built on Pocket-TTS. The goal was to clone the repo, build the Docker image, launch it, verify health, and smoke-test the API — all tracked in Gibber format.

What worked

Structured task tracking. Five tasks with dependencies, phases, status transitions — all machine-readable. The agent could parse its own task state without re-reading English prose.
Project dictionary. Adding §openvox, §tts, §docker-compose, §smoke-test to the extension dictionary kept the task files tight and unambiguous.
Session log. session.gibber captured decisions (Docker path, port 49112, CPU-only) in a form that would survive context compaction.
Density. The completed-tasks record is ~380 tokens in Gibber. The equivalent English Markdown would be ~900+ tokens. That's a 2.4x compression on structured state.

What didn't

Overhead on small tasks. For a 5-task install that took ~10 minutes, the time spent writing gibber files, updating the dictionary, and maintaining the index was disproportionate. A sticky note would have sufficed.
Template friction. The installed templates use placeholder values (TXXX, <iso-date>) that the agent had to read, understand, and replace. A cold-start cost.
No cross-session payoff yet. The real value of Gibber is persistence across sessions and compaction survival. A single-session task doesn't exercise that. The investment pays off later — if there is a later.
Spec reading cost. The agent consumed ~5k tokens reading GIBBER-SPEC, DICTIONARY, and RULES. For a first session that cost is amortized by the fast-path version check in future sessions, but it's steep upfront.

Case Study 002

Building Jarvis: a full AI assistant with Gibber as the spine

A multi-phase Rust project building a voice-controlled AI personal assistant. 45 Gibber files, 12 research digests, 16 task files, 202 dictionary symbols, 2,500 lines of Rust — all managed through Gibber from day one. This is the project Gibber was designed for.

2026-04-11 Claude Opus 4.6 16 tasks Long session, multi-phase Rust + Apple Silicon 45 .gibber files

The project

Jarvis is a voice-controlled, conversational AI personal assistant written in Rust. It receives voice commands, controls the computer, writes code, has persistent memory, and is extensible via MCP. The memory subsystem alone required deep research across vector databases, knowledge graphs, wiki-graphs, and document indexing — 8 parallel research agents producing findings that were immediately compressed into Gibber digests for the AI to reason over.

What worked

Research digests as dense knowledge. 8 research agents ran in parallel, each producing massive findings. Compressing each into a Gibber §digest form meant the AI could hold 12 research papers' worth of knowledge in ~12K tokens instead of ~60K+ in English. That's 5x compression on research data.
Learning log prevented repeated mistakes. The agent recorded 7 lessons (L001-L007) as structured Gibber entries: "cozo doesn't compile," "no Python," "always check deprecation first." These are machine-parseable — a future agent can read them and avoid the same errors without re-learning from conversation.
Dictionary extension scaled naturally. Started with 0 project symbols, grew to 202 organically as research domains expanded (vector DBs, knowledge graphs, wiki-graphs, document indexing, voice, computer control). Each domain added its symbols before using them — no undefined symbols, no ambiguity.
Synthesis form unified research. The §synthesis form (SYN001.gibber) cross-referenced all 12 digests into one architectural recommendation. 11 key insights, a final tech stack, a build order, and performance targets — all in ~2K tokens.
Parser dogfooding found real bugs. Building a Rust parser for Gibber (gibber-parse, 38 tests) and running it against all 45 .gibber files found patterns the spec didn't anticipate: dates as identifiers, §"quoted symbols", compound operators like §p-value:<0.001, and numbers with hyphenated units. The spec evolved from contact with reality.
Architecture decisions tracked as structured data. When Cozo failed to compile, the agent updated MEMORY.gibber with the new architecture, recorded the lesson in learning.gibber, and updated the task index — all as machine-readable Gibber forms. No prose to re-parse later.

What didn't

Agent jumped to code before setting up Gibber. The AI started writing Cargo.toml before creating task files, session logs, or memory entries. The user had to correct this twice. Lesson recorded as L001. The protocol didn't prevent the agent from ignoring it.
Spec doesn't cover all real patterns. The formal Gibber spec says symbols are "lowercase ASCII," but real usage needed uppercase (JARVIS-IDENTITY), dates (2026-04-11), timestamps, positive numbers (+7.9), and quoted symbols (§"hippocampus for AI"). The parser had to be extended beyond the spec.
Digest density has a readability floor. Some research digests became so dense they were hard even for the AI to navigate. D003.gibber (knowledge graphs, ~9KB) packs 15 database comparisons with benchmarks, licenses, and features into nested S-expressions. Dense is good; impenetrable is not.
No validation tooling yet. The agent wrote .gibber files freehand. Several had patterns the parser couldn't handle (found during round-trip testing). A gibber validate command run at write-time would have caught these immediately.
Task index maintenance is manual. Every time a task was added, the agent had to manually update tasks_index.gibber to add the new ID to the right phase. This should be automated.

Bottom line for Jarvis: Gibber was net-positive and dramatically so. The 5,000-token spec-reading cost was amortized within the first two research digests. By digest #12, the system had saved ~68,000 tokens compared to English — that's context budget the AI used for actual reasoning instead of re-reading verbose prose. The research-digest pattern (§digest) proved to be Gibber's killer use case: compressing complex multi-source research into dense, structured, cross-referenceable forms that survive compaction and can be synthesized. The learning log (§learning) prevented at least 2 repeated mistakes during the session. This is the project type Gibber was built for: long-running, multi-phase, research-heavy, with real architectural decisions to track.

Honest verdict

When Gibber earns its keep

Two case studies, opposite results. OpenVox (small, single-session) was net-negative. Jarvis (large, multi-phase, research-heavy) was net-positive by ~68K tokens. The pattern is clear.

Gibber shines when

  • ✓ Tasks span multiple sessions and context compaction is likely
  • ✓ Multiple agents need to read each other's state (multi-agent)
  • ✓ Task graphs have 10+ nodes with real dependencies
  • ✓ The project has a recurring domain vocabulary worth naming once
  • ✓ Session logs need to survive compaction as structured data, not prose

Gibber is overkill when

  • ✗ The task is short and single-session (like this one)
  • ✗ No compaction will happen before the work is done
  • ✗ The human needs to read every intermediate file
  • ✗ There are fewer than 5 tasks with no dependencies
  • ✗ The project will never be revisited in a later session

Bottom line for OpenVox: Gibber was net-negative for this specific task — the install was too small and too fast to justify the setup cost. But the infrastructure is now in place: the dictionary, the templates, the session log. If the project grows (voice cloning, API integration, multi-agent pipelines), the next session starts at zero spec-reading cost and can parse its own history. The investment is front-loaded; the return is cumulative.

Methodology

How we measure

Every case study follows the same protocol to keep comparisons fair.

1

Same task, measured both ways

We record the actual Gibber files the agent wrote, then estimate the English equivalent by expanding the Gibber forms using the render-english rules from GIBBER-RULES.md.

2

Token counts, not word counts

We count tokens (cl100k_base) because that's what matters for context budgets. Word counts hide the density difference because Gibber symbols tokenize efficiently.

3

Honest about costs

We report the one-time spec-reading cost alongside the per-file savings. If Gibber doesn't break even in a given project, we say so. No cherry-picking.
