gibber/3 — complete reference

The Gibber Specification.

Version gibber/3 — the complete grammar, file model, and validation rules for the Gibber symbolic format.

Motivation

Why this exists

Gibber is a compact symbolic file format for AI agent project files. Files written in Gibber are designed to be read by AI agents, not by humans.

Speed and density for the AI

Agents read gibber files directly. The format is dense, low-token, and uses a closed vocabulary that the agent already knows. Long working sessions hold many more files in context.

Build for the AI, not the reviewer

As agents grow more capable and trusted, the percentage of time a human looks at intermediate working files drops toward zero. The right design assumption is that the human will not be reading these. We optimize for the agent.

Humans get English on demand

When a human wants to see the contents of a gibber file in plain English, the AI generates a readable view in the conversation, or writes a derivative .human companion file. The English view is generated from the gibber file, never the other way around.

Portable

The protocol is small enough to teach an AI agent in a single message. Works in any Claude Code project, in any agentic workflow, on any machine.

Architecture

File model

Gibber uses two file extensions with two clear roles. The .gibber file is the source of truth; the .human file is a derivative view.

.gibber — the source of truth

A .gibber file holds a single Gibber AST form, optionally with a tiny YAML frontmatter for the file id and dictionary version. It has no English section, no ## gibber heading, no ## human heading. It is just the Gibber form. This is the format the agent reads, writes, edits, and reasons about.

example: task T042 gibber 
---
id: T042
gibber_dict: meta/v1, eidetic_memory
---

(§task §id:T042
  §status:§wip
  §depends:[T041 T038]
  §owner:§ai
  §goal:(§build §glp §encoder §with:[§ast §round-trip §bench])
  §files:[crates/eidetic-glp/src/encoder.rs
          crates/eidetic-glp/src/dict.rs]
  §done:[§encode §decode §round-trip-test]
  §todo:[§load §dict §bench §integrate §verifier]
  §budget:(§time<2d §lat<1ms §p99<3ms §acc=100%)
  §tests:[§unit §round-trip §latency])

.human — derivative, generated on demand

A .human file is a plain English Markdown rendering of a .gibber file. It is generated by the agent, never authored by hand. It exists only when a human has asked to see one.

Rules for .human files

  1. The .gibber file is the authoritative source. The .human file is derived from it via the render-english rules in GIBBER-RULES.md.
  2. Agents never edit a .human file directly. To change content, edit the .gibber file and re-render.
  3. If a .gibber file is newer than its .human companion, the .human is stale and must be regenerated (or deleted) before showing it to a human.
  4. .human files are generally gitignored. Only .gibber files belong in version control.
  5. The agent generates .human views on demand during a conversation. No human-rendering step is built into any tool wrapper or file-write operation.

Compatibility note: Earlier versions of Gibber used a single Markdown file with ## gibber and ## human headings. That model is deprecated as of gibber/3. The two-file .gibber / .human model replaces it. Conversion is mechanical: keep the gibber section, drop the human section, rename from .md to .gibber, and remove the format: dual frontmatter line.

Language

Gibber-speak grammar

Gibber-speak is a small S-expression-style notation. The grammar is intentionally tiny so it can be parsed by ~100 lines of code in any language.

Tokens

Token type Rule Examples
Symbols Start with § followed by lowercase ASCII, digits, or hyphens §task, §wip, §glp-encoder, §p99
Identifiers Bare alphanumeric strings (no leading §) T042, encoder.rs
Numbers Bare digits, optional decimal point or unit suffix 2d, 15ms, 100, 0.95
Operators Comparison and equality < > = <= >=
Punctuation Nesting and lists ( ) [ ]
Field-value separator : immediately after a symbol binds a value to it §status:§wip

Structures

Structure Syntax Description
Atom §wip, T042, 15ms Any single token
List [a b c] An ordered, unnamed sequence
Form (§head a b c) A named structure where §head is the type and the rest are children
Field §key:value Binds a value (atom, list, or nested form) to a field
complete task example gibber 
(§task §id:T042
  §status:§wip
  §depends:[T041 T038]
  §owner:§ai
  §goal:(§build §glp §encoder §with:[§ast §round-trip §bench])
  §files:[crates/eidetic-glp/src/encoder.rs
          crates/eidetic-glp/src/dict.rs]
  §done:[§encode §decode §round-trip-test]
  §todo:[§load §dict §bench §integrate §verifier]
  §budget:(§time<2d §lat<1ms §p99<3ms §acc=100%)
  §tests:[§unit §round-trip §latency])

This is one form named §task with several fields. The whole structure is a tree. A parser can decode it into a typed struct in any language.

Whitespace and comments

  • • Whitespace is insignificant except as a token separator.
  • • Newlines and indentation are encouraged for readability but not required.
  • • Comments start with ; and run to end of line.

Principle

The single-source-of-truth rule

Every .gibber file is the only authoritative representation of its content. There is no second half, no parallel section, no dual-language pair to keep in sync.

1

The .gibber file is the source

Always. Edits go here.

2

.human files are derived

Generated on demand by the agent from the matching .gibber file via the render-english rules in GIBBER-RULES.md.

3

No drift is possible

Because there is only one editable artifact per piece of information.

4

Validation is single-source

A validator parses the .gibber file's form and applies the matching validate rule from GIBBER-RULES.md. There is no two-half comparison because there is no second half.

Vocabulary

Dictionaries

A gibber dictionary is a list of symbols and their meanings, written in English. Dictionaries are versioned. A file's frontmatter names which dictionary it uses, e.g. gibber_dict: meta/v1.

1

Meta-work dictionary

meta/v1 — for task files, learning logs, session logs, memory dumps. Project-agnostic. Lives in GIBBER-DICTIONARY.md.

2

Project-specific extensions

Each project may add domain symbols in its own gibber-dict-<project>.md file. The project's CLAUDE.md lists which extensions are loaded alongside meta/v1.

3

Specialized dictionaries

Runtime AI-to-AI compression, AST encoding for source code, etc. Reserved for tools that build on Gibber, not part of the meta-work standard.

Standard library

Built-in functions

Gibber includes a small standard library of functions the AI executes mentally on read. They are pattern-rewrite rules — no external code, no library to import, no runtime to install.

§rep — repetition

(§rep N X)  ->  [X X X ... X]   ; X repeated N times

; Example:
(§rep 3 §test)   ->   [§test §test §test]

§range — integer range expansion

(§range A B)   ->   [A A+1 A+2 ... B]

; Example:
(§range T001 T003)   ->   [T001 T002 T003]

§ref — cross-reference

(§ref ID)         ->  the structure named ID, in full
(§ref ID FIELD)   ->  the value of FIELD within ID

; Example (assuming T042 has §status:§wip):
(§ref T042)            ->  the entire T042 structure
(§ref T042 §status)    ->  §wip

A task index does not have to restate every task's full content. It can list IDs and use §ref to pull fields when needed.

§sub — substitution macro

(§sub NAME VALUE)

; Defines NAME as VALUE within the current file scope.
; Example:
(§sub §my-stack [§rust §kuzu §glp])
...
§my-stack   ->   [§rust §kuzu §glp]

§expand — symbol to English

(§expand §wip)   ->   "work in progress"

; Looks the symbol up in the loaded dictionary
; and emits its English meaning.

§compress — English to symbol

(§compress "work in progress")   ->   §wip

; The inverse of §expand.

§concat — concatenation

(§concat A B C)

; Joins atoms or lists.
; With strings: produces a single string.
; With lists: produces a single flattened list.

§lookup — dictionary definition

(§lookup §wip)   ->   "work in progress"

; Like §expand but always returns the string definition
; exactly as written in the dictionary.

Usage hints

  • • If you write the same symbol three or more times in a row, use §rep.
  • • If you list a contiguous range of IDs, use §range.
  • • If a task field would restate another task's content verbatim, use §ref.
  • • If a tuple or list appears two or more times in the same file, define it once with §sub.
  • • When writing the human half, expand all function calls into their resolved values.
  • • Never use a function the AI cannot execute from this spec alone. No escape hatch to external code.

Correctness

Validation rule

A gibber file is valid if and only if all five conditions are met. If any fails, the file is rejected and the agent must fix it before writing.

  1. 1. The frontmatter contains a gibber_dict entry naming the dictionary version.
  2. 2. The body contains exactly one top-level Gibber form (the AST root).
  3. 3. The gibber form parses cleanly under the named dictionary — every symbol is defined in the dictionary.
  4. 4. All nested forms and fields follow the grammar rules (balanced parentheses, valid field bindings, correct token types).
  5. 5. If a .human companion exists, a semantic comparison of the two finds no contradictions in ID, status, or key fields.

Workflow

How an agent uses gibber

Four steps define the agent's interaction model with gibber files.

1

Reading a gibber file

Parse the gibber form into a typed structure. Work with the structure directly. Skip any .human companion — it is just a view.

2

Writing or updating

Produce the gibber form first, then write the .gibber file. Generate a .human companion only if a human asks.

3

Encountering a new symbol

Check the loaded dictionary. If the symbol is not present, add it to the appropriate extension dictionary with a clear English definition before using it. Never use undefined symbols.

4

Detecting drift

If the .gibber and .human files disagree, stop, fix the disagreement (always by updating the .gibber file and re-rendering), then proceed.

The bigger picture

Why this matters

In a long-running agent project, internal documents — task lists, learning notes, session logs, memory dumps — accumulate fast. They eat context budget. They slow generation. They make session compaction more lossy.

Gibber addresses all three. Dense gibber files stay in the agent's context at a fraction of the token cost. English views are generated on demand for the human reviewer. Both perspectives exist without a translation step or synchronization burden.

This is dogfooding compression at the meta-work layer of the project itself.

The entire specification, plus the standard meta-work dictionary, plus the templates, fits in a few small files. Designed to be copied into any Claude Code project with one install command, pasted into any agent's CLAUDE.md so the agent learns gibber immediately, shared with another developer as a single archive, and extended per project by adding a small domain-specific dictionary. Any agent in any project can speak gibber from its first message, with no per-project re-teaching.
