Gibber Tools.

Compress CLI output before it hits your agent's context. 1,500 tokens becomes 60.

The bottleneck no one talks about

The problem.

Every time your AI agent runs a CLI command, the raw output floods into conversation context. Most of it is noise. All of it costs tokens.

100% noise, 5% signal. The agent pays for every banner, progress bar, ANSI escape code, hint line, and timestamp. It uses maybe 5-20% of the content — the actual answer it needed.
A single build = 2,000 tokens. One cargo build can spend 2,000 tokens to communicate a single fact: "it compiled" or "line 42 has a type error." That is an absurd token-to-information ratio.
Tool output is the silent context killer. Over a long coding session, tool output is often the single largest consumer of the context budget. Not your code. Not your prompts. The output of git status, docker ps, npm install.
Every wasted token displaces reasoning. Context is a fixed budget. Every token the agent spends re-reading compiler banners is a token it cannot spend thinking about your architecture, your bug, your next step. The waste compounds across every command in the session.

How it works

The fix.

Every compressible CLI tool gets a small rule file that teaches Gibber how to extract the signal and discard the noise.

1

Invoke correctly

The rule specifies how to run the tool with its most structured output mode — e.g. cargo build --message-format=json.

2

Parse to AST

Raw output is parsed into a traversable structure: JSON, line-based, regex, or Tree-sitter grammar.

3

Filter by profile

A named profile selects which AST nodes matter: "errors + summary" vs "full trace" vs "summary only."

4

Emit Gibber

The filtered AST becomes a compact Gibber form the agent reads for pennies. Structured, parseable, dense.

5

Render English

If a human asks to see the result, the Gibber form renders back to clean English prose. No information lost.

Two paths to compression

Two mechanisms.

Pick the one that fits your workflow. Both use the same rule files and produce the same compressed output.

opt-in ship today

Wrapper script: gibber-run

The agent runs gibber-run cargo build instead of cargo build. The wrapper reads the rule, invokes the tool, parses, filters, and emits Gibber. No harness changes needed.

syntax 
gibber-run [--raw] [--profile NAME] <command> [args...]

--raw       bypass filter, return raw output (debugging)
--profile   pick a named profile (errors, full, summary...)
automatic ship later

Claude Code Bash hook

A post-Bash hook in ~/.claude/settings.json intercepts every Bash tool call, looks up the command in the rules directory, and rewrites the output through the same filter. Automatic for every command — the agent never needs to remember a wrapper.

how it works 
1. Agent calls Bash("cargo build")
2. Hook intercepts, finds cargo.rules
3. Rewrites output through the filter
4. Agent receives compressed Gibber form

The schema

Rule file format.

Each rule file is itself a Gibber form in tools/rules/<tool>.rules. One file per tool. Here is the schema.

§tool-rule schema gibber 
(§tool-rule §tool:<tool-name> §version:<version>
  §invoke:(§cmd <argv-template> §output:§json | §lines | §raw)
  §parse:(§format §json | §regex | §lines
          §schema:<schema-spec>)
  §profiles:[
    (§profile §name:<name>
      §keep:[<node-selector> ...]
      §drop:[<node-selector> ...]
      §render:(<gibber-template>))]
  §default-profile:<name>
  §render-english:(<template>))

§invoke

Specifies how to actually run the tool. May include flags added automatically to get structured output (e.g. --message-format=json).

§parse

How to turn raw output into a traversable structure. For JSON the schema is implicit; for line-based or regex parsing it names the fields per line.

§profiles

Named filters. Each profile says which AST nodes to keep, which to drop, and how to render the keepers as Gibber. Multiple profiles per tool.

§default-profile

Used when no --profile flag is passed. Typically the most useful balance of signal and brevity for day-to-day work.

§render-english

Used when the user wants to see the filtered result in human prose instead of Gibber. Lossless round-trip from the compressed form back to readable English.

§schema

Defines the expected structure of the parsed output. Implicit for JSON tools, explicit for line-based and regex parsers.

Extend it

How to add a new tool.

Each high-impact tool is 50-150 lines of rule definition. You write it once. Every future session benefits forever.

1

Create the rule file

Add tools/rules/<toolname>.rules following the §tool-rule schema above.

2

Extend the dictionary

Add any new symbols to tools/tools-dictionary.md with English definitions.

3

Capture fixtures

Record 3-5 real outputs (success, warning, error) in tools/fixtures/<toolname>/.

4

Run conformance

Execute gibber-run --selftest <toolname> to verify every load-bearing fact is extracted.

5

Done

The agent can now run gibber-run <toolname> [args] and get compressed structured results.

Eyes open

Honest caveats.

Gibber Tools is not magic. Here is what you should know before adopting it.

Format changes break rules

Tool output formats change with upstream versions. When cargo updates its JSON schema, rule files need updating. The conformance harness (capture real outputs, compare against expected) catches regressions — but someone still has to fix them.

Some tools are messy

Interactive tools, animated progress bars, tools that use ANSI for layout — these do not have clean parseable output. For these, generic.rules strips ANSI codes and common stopwords as a best-effort fallback. It helps, but it is not surgical.

Sometimes you want raw

Debugging unusual failures, investigating a tool's behavior, reading verbose logs on purpose. The --raw flag and the --profile full profile always provide an escape hatch. Compression is the default, not a prison.

Writing rules is real work

Each high-impact tool rule is 50-150 lines. You have to understand the tool's output format, decide which fields matter, and write the Gibber template. But you pay this cost once per tool, and every future session benefits forever.
